  **结论**
  可以替掉 `crawl4ai + NER` 这一层。
  不建议替掉 `DB-first authoritative shortlist` 这一层。
  也不建议把 Jina 直接拿来做“全库找同名 POI”的唯一手段。

  **为什么我认为它比现在更好**
  1. 现在真正的问题不是模型不够强，而是“比较单位错了”。
  你当前很多时候在用“网页标题/摘要”去对“POI 实体”，这不是 embedding 最擅长的任务。Jina 很强，但它也解决不了“榜单页 snippet 和单个 POI 是否同一实体”这种 page-level 对齐问题。

  2. `Tavily Extract` 天然更适合做“网页内容清洗层”。
  官方文档说明它支持从 URL 列表批量提取清洗内容，最多 20 个 URL，一次请求里还能用 `query` 做 chunk rerank，用 `chunks_per_source` 控制每个网页只返回最相关的几个片段。
  来源：
  [Tavily Extract docs](https://docs.tavily.com/documentation/api-reference/endpoint/extract)
  [Best Practices for Extract](https://docs.tavily.com/documentation/best-practices/best-practices-extract)

  3. 这会直接减少你现在的两个脆弱点：
  [Crawl4aiClient.ts](D:/AAA_Edu/geoloom-agent/backend/src/skills/web_poi_discovery/crawl4aiClient.ts)
  [NerClient.ts](D:/AAA_Edu/geoloom-agent/backend/src/skills/web_poi_discovery/nerClient.ts)

  4. LLM 提取 mention，比 LTP NER 更适合榜单页、攻略页、标题党页面。
  尤其是景点/美食这类网页，经常是：
  “老通城、四季美、江汉关、黎黄陂路……”
  传统 NER 和 regex 很容易抽脏、抽泛词、抽残词。
  LLM 用结构化 schema 抽 mention，通常更稳。

  **但你这个方案要改一处关键思路**
  不要“抽纯净正文后自己再大切块”。
  更好的做法是直接利用 Tavily Extract 自带的 `query + chunks_per_source`。

  原因：
  1. 官方文档说带 `query` 时，它会按相关性重排 chunk。
  2. `chunks_per_source` 每源最多 1 到 5 个 chunk，天然帮你控上下文。
  3. 这样你给 LLM 的不是“整页正文”，而是“和当前问题最相关的局部片段”。

  这一步会明显更快，也更便宜。

  **为什么你现在会觉得‘按理应该很快’但实际上没有**
  不是只有 crawl4ai 慢，当前总时延是几段叠起来的：

  1. Tavily 现在是顺序跑多条 query，不是并发。
  [TavilySearchSkill.ts](D:/AAA_Edu/geoloom-agent/backend/src/skills/tavily_search/TavilySearchSkill.ts)

  2. 本地语义重排本身就有 1s 级耗时。
  你的日志里已经能看到 1.5s 左右。

  3. Jina embedding + reranker 是远程调用，不是本地零成本函数。
  网络 RTT、批量大小、串行 orchestration 都会放大延迟。

  4. 当前 `entity_alignment` 还把“身份判定”和“推荐排序”混在一起。
  [EntityAlignmentSkill.ts](D:/AAA_Edu/geoloom-agent/backend/src/skills/entity_alignment/EntityAlignmentSkill.ts#L513)
  这里 identity confidence 直接混了 `name + spatial + category`。
  这会让向量模型在错误任务上消耗算力。

  **所以新方案怎么落，才会又快又准**
  我建议链路改成下面这样：

  1. 先做 query type / scope 判定
  `汉口景点推荐` 先判成 broad regional recommendation，不要走普通 nearby lookup。

  2. 本地 DB 先召回 authoritative shortlist
  按 scope + category 拉 50 到 200 个本地候选。
  这是权威集合，后面所有 web mention 只允许往这里对。

  3. Tavily Search 只保留 1 到 2 条 query
  第一条必须是用户原句。
  第二条最多是 shortlist 的分组验证句。
  不要扩成 3 条以上串行 query。

  4. 对 top 4 到 6 个 URL 调 Tavily Extract
  参数建议：
  `extract_depth=basic`
  `format=markdown`
  `query=用户原句或验证句`
  `chunks_per_source=2 或 3`
  这样你拿到的是“相关片段”，不是整页噪声。

  5. 用小模型并发提取 mention
  每个 chunk 提一个结构化 JSON：
  `mention`
  `evidence_span`
  `page_title`
  `url`
  `area_hint`
  `category_hint`
  `confidence`
  `is_generic`
  重点是让 LLM 抽“提及片段 + 证据”，不要只返回名字。

  6. mention 聚合与归一化
  把“老通城豆皮”“老通城酒楼”“老通城”归并。
  过滤泛词、场景词、文章小标题、区域词。

  7. 匹配时只在 local shortlist 内做
  这一步才让 Jina 发挥：
  先 lexical gate
  再 vector recall
  最后 reranker disambiguation
  不要对整库做开放式向量匹配。

  8. 输出分桶
  `verified_db_pois`
  `db_only_pois`
  `web_unresolved_mentions`
  地图/标签云只吃前两类。

  **这套方案比现在强在哪**
  1. 去掉了 `crawl4ai + NER` 两个不稳定服务点。
  2. 网页侧输入从 page-level 变成 mention-level。
  3. LLM 负责“抽提及”，Jina 负责“候选判别”，职责更清晰。
  4. 全程保持 DB-first，不会把网页里脏实体直接带进地图。

  **这套方案的真实风险**
  1. 如果你还是对“整页全文”做 LLM 分块，会更慢更贵。
  2. 如果你让 LLM 直接从网页里“发现主 POI”，还是会漂。
  3. 如果你还让 Jina 对全库做全局匹配，还是会语义相近但实体错误。
